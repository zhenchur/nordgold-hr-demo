# Fix List: Reload & Page Entry Behavior

> This list is for sequential fixes. Mark completed items as `[x]`.

## 1) Initial audit findings (до уточнений)

1. [x] Убрать автоматическое восстановление скролла на `reload`/`back_forward` если продукт должен стартовать с первого экрана.
2. [x] На перезагрузке не использовать `settleAtCurrentScroll` как базовый сценарий для шторки.
3. [x] Не завязывать первичный запуск intro только на одном флаге типа `shouldPlayInitialHeroIntro`.
4. [x] На `pageshow` корректно пересинхронизировать `ScrollTrigger` и системы reveal.
5. [x] После первого `refresh` дополнять прогон `replayVisible`, чтобы элементы не теряли запуск из-за layout race.

## 2) Уточнения от пользователя и итоговая цель

1. Поведение должно быть как на production-подходе, без “продолжения” состояния после reload.
2. Режим запуска по умолчанию — повторяемый и понятный: всегда старт с первого экрана.
3. Сохранение скролла между сессиями не используется для этого кейса.
4. При переходах назад/вперёд (стрелки браузера) — всегда показывать первый блок и запускать intro-зону.
5. Deep-link / hash-логику приоритетно не используем.

## 3) Что уже применено сейчас (стабилизированный baseline)

1. [x] `src/app/pageScrollRestoration.ts`
   - `restore()` всегда делает `scrollTo(0, 0)`.
   - Базовый режим фиксации скролла переведен в “top-поведение”.

2. [x] `src/main.ts`
   - Инициализация опыта приводит к `heroShutter.playMask(true)` после `restore()` и `ScrollTrigger.refresh()`.
   - Убран путь с `settleAtCurrentScroll` на первичном запуске.
   - На `pageshow` с `persisted` запускается форсированный повторный сценарий (force mode).

3. [x] `pageScroll.restore()` больше не нацелен на восстановление “старой позиции” для старта анимаций.
